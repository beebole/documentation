## Introduction

Beebole's migration guide helps existing customers understand the new platform, run a free trial, and decide when to move their account. It explains how the legacy system and the new system coexist, what features are available, and how the new API and architecture changes affect your planning.

<Info>
The legacy Beebole system remains active with no disruption to your current usage or data. You can create a free trial in the new platform while your legacy account continues to work.
</Info>

## What the new platform means for existing customers

When you try the new Beebole platform, you are evaluating the next-generation experience without switching your live account immediately. The new platform is a separate system that can run alongside your legacy account.

- You can create a free trial account in the new system and explore it independently.
- Your legacy data stays available until we manually migrate your account.
- We add companies to the migration list one by one, so contact us when you are ready.
- This is a chance to prepare your team, projects, tasks, and integrations before the manual transition.

<Info>
The migration itself is handled by Beebole support. Historical projects, people, and time records are migrated manually on a case-by-case basis.
</Info>

## Rethink your architecture for the new platform

The new platform is a good moment to revisit how your organisation uses projects, tasks, teams, tags, and custom fields.

- Use tags to model departments, locations, cost centers, or cross-functional teams.
- Use custom fields to capture information that does not fit in built-in project, person, or task fields.
- Keep project categories for structural project hierarchy, and use tags for reporting and availability.
- Assign schedules and rates through tags where possible, then override at the person or project level only when needed.

<Info>
A simpler structure is usually easier to maintain. Start with the minimum tag tree and add complexity only where it improves reporting or availability.
</Info>

### What to review before you migrate
- Your current project hierarchy and whether it still reflects how work is delivered.
- The task workflow and whether task statuses or dependencies should be simplified.
- How people are grouped for reporting and approvals.
- Any custom fields you need on projects, people, tasks, or time entries.

## Create a trial account and explore the new platform

Customers who want to try the new system can create a free trial account without affecting the legacy system.

- Sign up for a free trial in the new platform.
- Use the trial account to explore the dashboard, reports, tasks, and the GraphQL API.
- Keep the legacy system open while you trial the new system.

<Info>
The new trial account can exist at the same time as your existing legacy account. There is no need to close or migrate the legacy account until your company is ready.
</Info>

## Logging in when both systems exist

The new and legacy systems share the same login page. Beebole will detect which system an account belongs to and send you to the right experience.

- If your email is only registered in the legacy system, you go to the legacy account.
- If your email is only registered in the new platform, you go to the new account.
- If you have both accounts, you get to choose which system to access.

<Info>
This makes it easy to switch between the two systems during evaluation and migration planning.
</Info>

## New API and developer readiness

The new platform includes a GraphQL API for querying and mutating Beebole data. This is the primary developer-facing change for customers with integrations or automation.

- Review `/help/api/introduction` for the API basics and authentication model.
- Use the API to build or update integrations for projects, people, tasks, time records, absence records, and custom fields.
- Plan the migration of any legacy integrations as part of the manual account migration.

<Info>
The API is a major reason to evaluate the new platform early. It is also the right time to document your current integration points and decide which data should move first.
</Info>
