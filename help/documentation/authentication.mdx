Beebole supports multiple authentication methods so your team can sign in securely and conveniently. You can use a combination of methods depending on your organization's security requirements.

---

## Sign-in methods

### Email and password

The default sign-in method. Each person creates a password when they accept their invitation to Beebole.

### Google and Microsoft sign-in

Team members can sign in using their existing Google or Microsoft account. This is especially convenient for organizations already using Google Workspace or Microsoft 365.

<Info>
  OAuth sign-in uses the email address associated with the Google or Microsoft account. The person's Beebole profile must use the same email address.
</Info>

### Passkeys

Beebole supports passwordless sign-in with passkeys — using fingerprint, face recognition, or a hardware security key. Passkeys work across all your devices and provide a fast, phishing-resistant sign-in experience.

To set up a passkey, go to your profile settings and select **Security** > **Add passkey**.

### PIN-based sign-in

For quick access, Beebole offers PIN-based authentication. Each person can set a personal PIN code for faster sign-in on trusted devices.

## Single sign-on (SSO)

For organizations that need centralized authentication, Beebole supports SSO via OpenID Connect. This lets your team sign in with your existing identity provider (e.g., Okta, Azure AD, Google Workspace).

<Steps>
  <Step title="Open account settings">
    Go to **Settings** > **Account**.
  </Step>
  <Step title="Configure SSO">
    In the **Single sign-on** section, enter your OpenID Connect configuration details: issuer URL, client ID, and client secret.
  </Step>
  <Step title="Test the connection">
    Use the test option to verify that sign-in works correctly before enabling SSO for your entire team.
  </Step>
  <Step title="Enable SSO">
    Once tested, enable SSO. Team members will be redirected to your identity provider when signing in.
  </Step>
</Steps>

<Warning>
  When SSO is enabled, make sure at least one administrator retains email/password access as a backup in case the identity provider is unavailable.
</Warning>

## API keys

Developers who integrate with Beebole's [GraphQL API](/api/introduction) authenticate using API keys.

- API keys can be generated in your profile settings under **Security** > **API keys**.
- Each key has a configurable expiration date.
- Treat API keys like passwords — do not share them or commit them to version control.

---

## Account deletion

You can permanently delete your Beebole account and all associated data from your profile settings. This action is irreversible.

<Warning>
  Account deletion removes all your data — time entries, projects, people, and configuration. This cannot be undone. Export any data you need before proceeding.
</Warning>

## Frequently asked questions

<AccordionGroup>
  <Accordion title="Can I use multiple sign-in methods at the same time?">
    Yes. A person can have an email/password, a linked Google or Microsoft account, and a passkey — all active simultaneously. They can use whichever is most convenient.
  </Accordion>
  <Accordion title="Does SSO replace other sign-in methods?">
    When SSO is enabled, it becomes the primary sign-in method. However, administrators can retain email/password access as a backup.
  </Accordion>
  <Accordion title="Can I require all team members to use SSO?">
    Yes. Once SSO is configured, you can enforce it as the only sign-in method for your organization.
  </Accordion>
  <Accordion title="How do I reset a team member's password?">
    As an administrator, you can resend an invitation from the person's profile, which lets them set a new password. Alternatively, the person can use the "Forgot password" flow on the sign-in page.
  </Accordion>
  <Accordion title="What happens to API keys when a person is archived?">
    API keys belonging to archived users are deactivated. If you reactivate the person, you'll need to generate new keys.
  </Accordion>
</AccordionGroup>
