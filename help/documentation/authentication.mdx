Beebole supports multiple authentication methods so your team can sign in securely and conveniently. You can use a combination of methods depending on your organisation's security requirements -- from simple email and password to enterprise SSO with OpenID Connect.

---

## Sign-in methods

### Email and password

The default sign-in method. Each person creates a password when they accept their invitation to Beebole. Passwords must meet minimum security requirements and can be changed at any time from the person's profile.

### Google and Microsoft sign-in

Team members can sign in using their existing Google or Microsoft account. This is especially convenient for organisations already using Google Workspace or Microsoft 365.

<Info>
  OAuth sign-in uses the email address associated with the Google or Microsoft account. The person's Beebole profile must use the same email address.
</Info>

### Passkeys

Beebole supports passwordless sign-in with passkeys -- using fingerprint, face recognition, or a hardware security key. Passkeys work across all your devices and provide a fast, phishing-resistant sign-in experience.

<Steps>
  <Step title="Open your profile">
    Click your profile avatar, then select **Profile**.
  </Step>
  <Step title="Go to the security section">
    Select the **Security** tab.
  </Step>
  <Step title="Add a passkey">
    Click **Add passkey** and follow your browser's prompts to register a fingerprint, face, or hardware key.
  </Step>
</Steps>

<Tip>
  You can register multiple passkeys -- for example, one on your laptop and another on your phone -- so you always have a way to sign in.
</Tip>

### Email code (passwordless)

Instead of entering a password, you can sign in with a one-time verification code sent to your inbox. On the sign-in page, click **Send code instead** and Beebole sends a 6-digit code to your email address. Enter the code to complete sign-in.

<Info>
  The code is single-use and expires after a short time. If you do not receive it, click **Didn't receive the code? Resend** to request a new one.
</Info>

### PIN-based sign-in

For quick access, Beebole offers PIN-based authentication. Each person can set a personal PIN code for faster sign-in on trusted devices. This is useful in shared environments like workshops or kiosks where speed matters.

To set up a PIN, go to your profile and look for the **PIN** option in the **Security** section.

## Single sign-on (SSO)

For organisations that need centralised authentication, Beebole supports SSO via OpenID Connect. This lets your team sign in with your existing identity provider. Beebole supports three SSO providers: **Google**, **Microsoft**, and a **Custom OpenID** provider (e.g., Okta, Auth0, Entra ID).

SSO is available as part of the **Custom Domain & SSO** add-on. Go to **Settings** > **Single Sign On** to access the configuration.

### Google SSO

<Steps>
  <Step title="Open SSO settings">
    Go to **Settings** > **Single Sign On** and select the **Google** tab.
  </Step>
  <Step title="Link your domain">
    Enter your organisation's email domain (e.g., `example.com`) and click **Add**. Once a domain is linked, users are redirected to Google sign-in when they enter a matching email address.
  </Step>
  <Step title="Enable auto-provisioning (optional)">
    Turn on **Automatically create new users on first SSO login** to provision accounts automatically when a user with a matching domain signs in for the first time.
  </Step>
  <Step title="Enforce Google sign-in (optional)">
    Enable **Disable interactive sign-in. Only Google sign-in allowed** to block password and email-code sign-in for your organisation.
  </Step>
</Steps>

### Microsoft SSO

<Steps>
  <Step title="Open SSO settings">
    Go to **Settings** > **Single Sign On** and select the **Microsoft** tab.
  </Step>
  <Step title="Enforce Microsoft sign-in (optional)">
    Enable **Disable interactive sign-in. Only Microsoft sign-in allowed** to require all users to authenticate via Microsoft.
  </Step>
</Steps>

### Custom OpenID Connect

Use this option for any OpenID Connect-compatible provider (Okta, Auth0, Microsoft Entra ID, etc.).

<Steps>
  <Step title="Open SSO settings">
    Go to **Settings** > **Single Sign On** and select the **Custom OpenID** tab.
  </Step>
  <Step title="Enter your provider details">
    Fill in the **Issuer URL**, **Client ID**, and **Client Secret** from your identity provider. If your provider does not support OpenID Connect discovery, enable the **Other provider** toggle and enter the **Authorization URL** and **Token URL** manually.
  </Step>
  <Step title="Configure your provider">
    Copy the **Initiate URI** and **Callback URI** shown in Beebole and enter them in your identity provider's application settings.
  </Step>
  <Step title="Enforce custom SSO (optional)">
    Enable **Disable interactive sign-in. Only this SSO provider allowed** to block all other sign-in methods.
  </Step>
</Steps>

### Enforcing SSO organisation-wide

Each SSO provider has a dedicated enforcement toggle that disables interactive sign-in (email/password and email codes) for your entire organisation. Once enabled, users must sign in through the configured SSO provider.

<Warning>
  Before enabling enforcement, verify that SSO is working correctly by signing in as a test user. If your identity provider becomes unavailable, users will be locked out. Keep an administrator's direct access available as a fallback, or disable enforcement temporarily from the SSO settings.
</Warning>

## API keys

Developers who integrate with Beebole's [GraphQL API](/help/api/introduction) authenticate using API keys.

<Steps>
  <Step title="Open your profile">
    Click your profile avatar, then select **Profile**.
  </Step>
  <Step title="Go to the security section">
    Select the **Security** tab.
  </Step>
  <Step title="Generate an API key">
    Click **Add API key**. Set an expiration date and click **Create**. Copy the key immediately -- it is only shown once.
  </Step>
</Steps>

- Each key has a configurable expiration date.
- You can have multiple active API keys.
- Treat API keys like passwords -- do not share them or commit them to version control.

<Warning>
  If an API key is compromised, delete it immediately from your profile and generate a new one.
</Warning>

## Admin takeover

Administrators can sign in as another person in the organisation to troubleshoot issues, verify settings, or assist with configuration. This is called admin takeover.

<Steps>
  <Step title="Open the person's profile">
    Go to **People** and select the person you want to sign in as.
  </Step>
  <Step title="Use the takeover option">
    Click **Sign in as this person**. You are now viewing Beebole exactly as that person sees it.
  </Step>
  <Step title="Return to your account">
    Click **Back to my account** in the notification bar at the top of the screen to return to your own session.
  </Step>
</Steps>

<Info>
  Admin takeover actions are recorded in the [audit trail](/help/documentation/audit-trail). The log shows that the action was performed by the administrator, not the person being impersonated.
</Info>

## Session management

Beebole manages active sessions to keep your account secure. Each time you sign in from a new device or browser, a session is created.

- Sessions expire after a period of inactivity.
- You can view your active sessions from your profile under the **Security** tab.
- Sign out of individual sessions or all sessions at once if you suspect unauthorised access.

---

## Account deletion

You can permanently delete your Beebole account and all associated data from your profile settings. This action is irreversible and removes your personal data from the platform.

<Warning>
  Account deletion removes all your data -- time entries, profile information, and configuration. This cannot be undone. Export any data you need before proceeding.
</Warning>

To delete your account, go to your profile, scroll to the bottom, and click **Delete my account**. Follow the confirmation prompts.

<Note>
  If you are the only administrator in your organisation, you must either transfer admin rights to another person or [delete the organisation](/help/documentation/account-settings) first.
</Note>

## Frequently asked questions

<AccordionGroup>
  <Accordion title="Can I use multiple sign-in methods at the same time?">
    Yes. A person can have an email/password, a linked Google or Microsoft account, and a passkey -- all active simultaneously. They can use whichever is most convenient.
  </Accordion>
  <Accordion title="Does SSO replace other sign-in methods?">
    When SSO is enabled, it becomes the primary sign-in method. However, administrators can retain email/password access as a backup.
  </Accordion>
  <Accordion title="Can I require all team members to use SSO?">
    Yes. Each SSO provider (Google, Microsoft, Custom OpenID) has an enforcement option that disables all other sign-in methods. Enable the **Disable interactive sign-in** toggle in the relevant tab under **Settings** > **Single Sign On**.
  </Accordion>
  <Accordion title="What is the email code sign-in option?">
    Email code sign-in lets you authenticate without a password. On the sign-in page, click **Send code instead** and Beebole sends a 6-digit code to your email address. This is useful if you have forgotten your password or prefer not to manage one.
  </Accordion>
  <Accordion title="How do I reset a team member's password?">
    As an administrator, you can resend an invitation from the person's profile, which lets them set a new password. Alternatively, the person can use the **Forgot password** flow on the sign-in page.
  </Accordion>
  <Accordion title="What happens to API keys when a person is archived?">
    API keys belonging to archived users are deactivated. If you reactivate the person, you need to generate new keys.
  </Accordion>
</AccordionGroup>
